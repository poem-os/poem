## Status

Review

## Story

**As a** prompt engineer,
**I want** to generate mock data matching the workflow-data schema,
**so that** I can test each prompt independently without real transcripts.

## Background

This story continues Epic 4: YouTube Automation Workflow (System Validation). Story 4.1 imported 53 YouTube Launch Optimizer templates, and Story 4.2 implemented schema extraction from those templates. Now we need to generate realistic mock data that conforms to the workflow-data schema so that each prompt can be tested independently without requiring real video transcripts.

**References**:
- Epic 4 Goal: `docs/prd/epic-details.md` (lines 435-451)

1. Mock data generated for all 37 workflow-data fields defined in spec

5. Mock `transcriptAbridgement` generates 500-1000 word summary text
6. Mock `createChapters` generates YouTube chapter format with timestamps
7. Generated mock data validates against extracted schemas
8. Mock data saved to `/poem/mock-data/youtube-launch-optimizer/workflow-data.json`

### Architecture Context: Data Models

[Source: docs/architecture/data-models.md § Schema]

**Unified Schema Structure**:

```typescript
interface UnifiedSchema {
  templateName: string;
  version: string;
  description?: string;
  input: {
    fields: SchemaField[];
  };
  output?: {
    fields: SchemaField[];
  };
}

interface SchemaField {
  name: string;  // Supports dot notation: "user.firstName"
  type: "string" | "number" | "boolean" | "array" | "object";
  required: boolean;
  description?: string;
  items?: SchemaField;          // For arrays
  properties?: SchemaField[];   // For objects
  constraints?: FieldConstraints;
}

interface FieldConstraints {
  min?: number;
  max?: number;
  minLength?: number;
  maxLength?: number;
  enum?: string[];
  fakerHint?: string;  // e.g., "person.firstName", "lorem.paragraph"
  pattern?: string;
}
```

**Key Principles**:
- Schemas behave like function signatures: `(input) -> output`
- Input section always required, output optional
- Dot notation for nested fields: `"object.field"` → needs nested object generation
- `fakerHint` provides guidance for realistic data generation
- `constraints` define boundaries for generated values
