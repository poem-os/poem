# Story 0.1: Work Intake Triage System

## Status

Done

## Story

**As a** POEM developer,
**I want** a unified work intake triage system,
**so that** I can quickly route development work to the appropriate workflow without decision paralysis or confusion.

## Background

**Problem**: POEM has 5+ different entry points for development work (existing stories, new stories in epics, quick fixes, Epic 0 stories, usage issues), but no unified way to decide which path to take. Developers experience decision paralysis when they have work but aren't sure whether it should be a story, a quick fix, Epic 0, or which epic it belongs to.

**Current Pain Points**:
1. Only existing stories have a clear path (`/appydave-workflow {story}`)
2. New work requires manual decision-making about:
   - Is this <1hr (Quick Fixes) or >1hr (Story)?
   - Which epic does this fit? Or is it Epic 0?
   - Should I use Bob? Or just fix it directly?
3. Usage issues in `usage-issues.jsonl` have no conversion mechanism to stories/fixes
4. No clear criteria or decision tree documented

**Example from Recent Work**:
User wanted to create "README Enhancement System" for Epic 1. Had requirements in conversation but no clear path to:
- Decide if it's Epic 1 Story 1.10 vs Epic 0 Story 0.1
- Know whether to use Bob *draft or another command
- Convert conversation requirements into story format
- Kickstart AppyDave workflow

**Solution**: Create `/triage` skill that analyzes context (conversation, usage issues, explicit description), evaluates against decision criteria (time, epic fit, impact), and routes to the appropriate workflow with clear recommendations and handoff context.

**Epic 0 Category**: Developer Experience (P1)
**Priority**: P1 (High) - Reduces friction in daily development workflow

### 2. Triage Task Creation (Core Logic)

1. **Time/Ceremony**: <1hr AND no ceremony needed ‚Üí Quick Fix, else Story

3. **Impact/Priority**: High impact justifies story-level tracking
4. **Type**: Bug/enhancement/refactor/docs helps determine Epic 0 vs Feature Epic

### Key Technical Decisions

1. **Why task + skill wrapper pattern?**
   - Task file (`.bmad-core/tasks/`) is source of truth (reusable by agents)
   - Skill wrapper (`.claude/commands/`) provides user-facing entry point
   - Multiple entry points, single source of truth

2. **Why top-level `.claude/commands/triage.md` not `/poem/`?**
   - Triage is about BMAD development workflow
   - POEM is the project being built with BMAD
   - `/poem/` directory is for POEM-specific features (Penny, Victor, log-issues)
   - Top-level is for general BMAD/development skills

3. **Why rename Maintenance Backlog to Quick Fixes?**
   - Avoids collision with Epic 0 "Maintenance & Continuous Improvement"
   - "Quick Fixes" clearly describes <1hr, no-ceremony work
   - Commands become clearer: `*add-fix` vs `*add-maint`

4. **Why include usage issue integration?**
   - Usage issues (`usage-issues.jsonl`) are already part of gap analysis workflow
   - Triage provides conversion mechanism: issue ‚Üí routed work
   - Single entry point for all work types

5. **What is the agent command format in handoffs?**
   - Format: "Next: /BMad/agents/sm then *draft" means TWO-STEP process:
     1. User loads agent: `/BMad/agents/sm` (activates Bob)
     2. User types command: `*draft` (Bob executes create-next-story task)
   - Triage DOES NOT auto-execute - it routes and hands off
   - User confirms and executes the suggested command sequence
