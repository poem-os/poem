## Overview

Assembles a complete YouTube video description from pre-generated components. This is a **pure assembly task** - all content has already been generated by upstream prompts. The output is formatted with YouTube-native syntax and ready to paste directly into YouTube.

**Purpose**: Combine first line, simple description, chapters, CTAs, social links, hashtags, and legal disclaimer into a single well-formatted YouTube description.

## Architecture Patterns

### 1. Pure Assembly Pattern

**Pattern**: This prompt does NO content generation. It only arranges pre-generated pieces.

**Why**: Clean separation of concerns - content generation is handled upstream, this prompt handles formatting and structure only.

**Contrast with v1**: v1 had generative language ("craft engaging and informative"). v2 explicitly states "PURE ASSEMBLY task" (all caps for emphasis).

**Related**: See `docs/architecture-decisions/srp-recommendation.md` - this is the assembly step in a multi-stage pipeline.

### 2. YouTube-Native Formatting Pattern

**Pattern**: Output uses YouTube's limited formatting syntax directly (no markdown conversion needed).

**Why**: YouTube's description field supports specific formatting (single asterisk for bold, `# Heading` for large headers, raw URLs auto-link). This prompt outputs in YouTube's native format for direct paste.

**Supported Formats**:
- `*text*` = bold (NOT `**text**` markdown)
- `# Heading` = large heading
- `## Heading` = medium heading
- `- item` = bullet point
- Raw URLs auto-link (no `[text](url)` needed)
- `0:00` timestamps auto-link to chapters

**Anti-Pattern**: v1 used generic "write the output in code block" which wasn't YouTube-specific.

### 3. Dynamic Section Ordering Pattern

**Pattern**: Chapters section can move from position 7 to position 8 based on length (>20 lines).

**Why**: Long chapter lists push CTAs below the fold. Moving chapters to end keeps CTAs visible.

**Implementation**: Conditional logic based on chapter count:
- Default: Chapters at position 7 (after social links, before hashtags)
- If >20 chapters: Move chapters to position 8 (just above hashtags)

**Human Checkpoint**: AI must count chapter lines and decide placement.

### 4. Emoji Header Scannability Pattern

**Pattern**: Each section has an emoji prefix for visual scannability.

**Why**: YouTube descriptions are long text blocks. Emoji headers help users quickly scan to find relevant sections (CTAs, chapters, social links).

**Emoji Map**:
- üëâ = Primary CTA
- üé• = Related Videos
- üîó = Tools & Affiliate Links
- üåê = Connect (Social Links)
- ‚è±Ô∏è = Chapters
- # = Hashtags

### 5. Brand Configuration Composition Pattern

**Pattern**: Brand-specific content (CTAs, affiliates, social links, legal) is referenced from `brandConfig` object.

**Why**: Enables brand reusability - same prompt works for multiple brands by swapping `brandConfig`.

**Brand Config Structure**:
```handlebars
brandConfig:
  ctas:
    primaryCta: {label, url}
  affiliates: [{active, name, url}]
  socialLinks: {website, twitter, youtubeMain, skool}
  descriptionTemplate:
    legalDisclosure: "string"
```

**Iteration Pattern**: Affiliates use `{{#each}}` + `{{#if active}}` to filter active links only.

### 6. Formatting Guidelines Pattern

**Pattern**: Explicit formatting rules to prevent over-decoration.

**Rules**:
- Use emoji headers for scannability
- Keep CTA format short: `üëâ Label: URL`
- Group all affiliate links under one heading
- One blank line between sections
- **No horizontal rules or decorative elements**

**Why**: YouTube descriptions should feel natural, not like formatted articles. Avoid markdown-style decoration.
